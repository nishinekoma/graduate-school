入学後の研究内容、方法（５００字程度）

入学後の研究
  はい、入学後の研究として偽装広告などのダークパターンが蔓延するWEBの環境に、ユーザーの認知負荷を軽減するための
  「自己防衛型ブラウザ拡張機能」の開発と評価をしようと考えています。従来のAdBlockなどの技術は、URLなどの静的構造に
  基づくパターンマッチングに依存しており、動的に変化する欺瞞的UIの検知には限界があります。対策としてクラウドAIなどを
  用いた動的解析がありますが、通信環境に依存するため、ＵＩ防御が間に合わなず誤操作させる可能性を孕んでいます。
  そこで本研究ではWebNNなどを用いたローカル推論によるUI防御を中心に、ユーザが誤操作を行う確率を低減するブラウザ拡張機能を構築します。
  特に研究を進める中で重要になるのが、「推論や描画に遅延が生じる状況でも、UI 防御が間に合う条件は何か」
  ということを明らかにする点です。研究方法として軽量化した判定モデルをブラウザ拡張機能として実装し、ユーザーがクリック操作する前に
  UI介入が成功したどうかの介入成功率を調べます。その後遅延による誤操作防止を実装した機能としないものを比べ、どのような改善が
  見られたかを研究します。

    →　つまり　推論が間に合わない状況でも、ユーザを誤操作させないUI防御方式を設計・提案する研究をして
    　　　　　　自己防衛型ブラウザ拡張機能として組み込み、それを導入して誤操作率はどう改善するのかを研究する。

話す内容
  ローカル推論でのダークパターンのUI防御が成立するための遅延条件を明らかにし、遅延しても防御できる方法を研究します。
  前提として先行研究では、ブラウザ内推論を行うとデバイスによってはOS上でモデルを実行する速度の
  約５倍から16倍になり、 モデルによっては多くのメモリを要求します。また推論後に
  WEBブラウザ内でGUIコンポーネントをレンダリングする時間を約67%増加させると研究結果が出ています。
  これは使用者のデバイススペックによっても推論速度やGUIレンダリングが遅れてユーザが意図しない操作を行ってしまう可能性があります。
  そこでまず、何ms以内であればUI介入が誤操作防止として有効なのかを明らかにし、その上で遅延を
  しても防御できる方法を研究してダークパターン防御が間に合うことを保証することを目指します。

ダークパターンとは何ですか？
  ウェブサイト上でサイト閲覧者の操作に対して意図しない行動をとらせるUI（ユーザインターフェース）
  のことです。パターンの一つとして有名なものだと実際のコンテンツと広告の境界を意図的に曖昧する、偽装広告というものがあります。
  例えばあるソフトウェアダウンロードのWEBサイトにて、希望するソフトウェアのダウンロードボタンと酷似した目立つ広告のダウンロード
  ボタンを表示させ、ユーザーを混乱させる手法などがあります。

ユーザが行う誤操作は何を想定していますか。
  はい、ユーザが意図した操作と、実際に発生するブラウザの挙動が食い違うことです。
  例えば、WEBサイトを閲覧していて、ユーザがサイトをスクロールしたりマウス移動などをしたとします。
  それがトリガーとなり、コンテンツに透明な広告が動的に挿入され、見た目は変わっていないのに、次のページを
  クリックした瞬間に別の広告サイトなどに飛ばされることを想定しています。

ローカル推論とは何ですか？
  外部のクラウドサーバにデータを送信せずに、自身のデバイスでAIモデルを動作させることです。
  例えばスマホのFaceIDによる画像認識や、Siriなどの音声認識などがあります。

なぜローカル推論で行おうと思ったのですか？
  はい、ダークパターンを防御する介入が早いと考えたためです。クラウド推論では、サーバにリクエストを送り結果を返される
  までの通信往復が、介入が間に合わない可能性があると判断しました。また、この通信往復の時間は、通信環境でもっと時間が
  延びる可能性があります。そのためこの通信往復を排除できるローカル推論であれば、そのような遅延が抑えられる可能性が
  あるため、ローカル推論で研究しようと考えました。

なぜ静的解析ではなく動的解析なのか。（みんな知ってるから出る可能性低い）

どのように実装し比較するのか。
  まずダークパターンによってユーザが誤操作をしてしまうのを、UI介入で防いで成功する時間や操作などの条件を整理して、
  それを満たすUI防御をブラウザ拡張として実装します。
  比較としては同じダークパターンに対して、実行構成や方式を変えて比較し、デバイスごとの性能を考慮して
  遅延があっても誤操作がどれだけ減るかを見ます。

実行構成や方式を変えて比較を具体的に
  WebNNを使用しない方法や、推論の実行タイミングを変えたりするなどのことを考えています。

なぜWebNNを用いるのか。
  ブラウザ内推論を高速に行えて、時間を削減できる可能性があるからです。
  WebNNを使う理由は、ブラウザ内推論をハードウェアに近い形で実行できる可能性があり、
  低遅延UI介入への適性を検証する価値があるからです。

  遅延対策の構成の一つとして、ブラウザ内推論を短くできれば、サイト内で検出した際にUI介入までの時間を一時停止などをした
  対策による心理的負担を減らすことがで切ると考えたためです。

WebNNとは？
    WebNNは、ブラウザ内での学習済みモデルを使って判断・予測する動作を
    効率的に実行できるようにした、Machine Learning専用に設計された低レベルのAPIです。
    具体的にはランタイム環境と、演算をするチップの間の低レベルンターフェースに当たります。
    NPUという推論に特化した演算処理チップを選択することができ、また計算負荷を最適化してから
    演算処理に投げるという利点もあるのが特徴です。
    
    NPUはAIの計算処理をやるための専用チップです。
    ランタイム環境＝ONNX Runtime Webなどのエンジンが、ブラウザ上でモデルを動かせる状態にしている場所

「推論や描画に遅延が生じる状況でも、UI 防御が間に合う条件は何か」
ということを明らかにする点といったが、なぜか。
  ローカル推論によるUI 防御は、デバイスの性能に依存するためです。
  例えばNPUなどのAI専用チップが使える端末では、高速に推論できるため
  ページを止めずに介入したほうが、ユーザの心理的負荷が小さくなります。
  逆に性能が低ければ、遅延が生じページを止める必要があります。
  そのため、「どこまで遅れても誤操作は防げるか」という条件を明らかにしないと、
  どの方法で介入をするべきかが判断できないからです。

    key ローカルでUI防御　デバイスの性能に依存　NPUなどのAI専用チップが使える端末　ユーザの心理的負荷が小さく。

モデルの軽量化方法
  ユーザの操作周辺だけを推論する形にする形で推論回数を減らしたりして、
  軽量化しようと考えています。

インターフェースとは？
「インターフェースとは、『異なる2つのものの間をつなぐ境界線や、情報の受け渡しルール』のことです。」
  たとば、一つのソフトを動かす際、メーカーによってOSの種類が別々ですが、そのソフトとOS間の受け渡し方を定義した
  ルールを設けたインターフェースを用いることによってその恩恵が受けることができます。

  従来の AdBlock 等の技術は URL や静的構造に基づくパターンマッチングに依存
    AdBlock 等の技術とは？
      Webブラウザの広告ブロック拡張機能のことを指す。
      AdBlockはグーグルの拡張機能のこと。
    
    静的構造に基づくパターンマッチングとは？
      ブラウザを構成するURLやDOM構造を解析して、広告やダークパターンに一致する文字列や属性を
      探し、リクエストや要素を遮断するフィルタリング技術のこと。

    静的パターン（Static Pattern）
      ページの状態やユーザー行動に関係なく
      読み込まれた時点（ロード中）のコード・URLだけで判断できるパターン

      静的構造で分かる事
        ・DOM ツリー
        ・属性
        ・URL 文字列

      静的構造ではわからないこと
        ・実行後の状態
        ・意味の変化
        ・意図の裏切り

    動的パターン（Dynamic Pattern）
      定義
        時間経過・ユーザー操作・文脈変化によって
        意味や役割が変わる UI 振る舞いのことを指す。

      詳細
        サーバが返すまで遷移先不明や
        日替わり地域・端末で変わるなど。
        遷移先がAPIサーバを通じて返される。
        元から決まっていなとか。

      動的でしかわからない事例
        ・遷移先がAPIサーバを通じて返される。
        ・ユーザー状態依存 UI
          初回訪問 → 記事を表示
          2回目以降 → 広告全画面表示
            依存情報
              Cookie
              LocalStorage
              ログイン状態
        ・

  ここでの自然言語処理（NLP）とは？
    自然言語処理（NLP）
      コンピュータが 人間の言葉（自然言語）を理解・解析・生成する技術です。
      機械学習を使用してコンピュータが人間の言語を理解する技術

      計算言語学（人間の言語ルール・ベースのモデリング）、統計モデリング、機械学習（ML）、
      ディープラーニングを組み合わせることで、コンピューターやデジタル・デバイスがテキストや音声を認識、理解、生成できるようにします。
    
    手法
      １．文章を単語や形態素といった「小さな単位」に分割する前処理。
          例
            「これはテストです。」 → ["これ", "は", "テスト", "です"]
        
      ２. 単語埋め込み（Word Embedding）
        単語をベクトル（数値ベクトル）として表現し、意味的な関係性を学習する方法。
          例
            king ≈ queen, man ≈ woman がベクトルの演算で意味的に近くなる

      ３．構文・意味解析（Parsing & Semantic Analysis）
          文の文法構造や意味を解析します。
            構文解析
            意味解析
            文脈依存関係解析

      ４．分類・ラベリング（Text Classification / Token Classification）
          文章や単語をカテゴリーに分類する技術。
            例
              ・ダークパターンの可能性が高い/低い
              ・インターフェイス説明文を意味カテゴリに分類

      ５．質問応答（Question Answering）
          文章から質問に対して答えを抜き出すモデル。
            例
              「この UI の説明文の要点は？」 のような意味抽出
      
      ６．言語生成（Natural Language Generation）
        意味理解だけでなく、反対に自然な文章を生成する技術。
        （チャットボット/要約など）
    学習方式
      ✔ 教師あり学習
        ・ラベル付きデータを使いモデルを訓練する方式
        ・分類/抽出などの精度が高い
        ・「これはダーク」「これは正常」とラベル付き
        ・精度は高い
        ・データ作成が大変

        例
          悪意ある UI テキスト vs 正常なテキストのラベルデータで学習する

      ✔ 教師なし学習
        ・ラベルがないデータを使い構造やパターンを学習する方式
        ・ラベルなしで構造を発見
        ・異常検知など
        ・解釈が難しい

      ✔ 自己教師あり（最近主流）
        ・大量テキストから自動で学習
        ・BERT系・LLM
        
        例
          クラスタリング、語義解析など

    自然言語処理を用いて心的パターンを抽出する
      例
        意味分散
          「登録内容を確認します」
          「次に進むことでサービスが有効化されます」
          特徴
            単語として「課金」「定期購入」は存在しない
            意味としては 課金確定
              ➡ 構文は安全、意味が危険

        Yes/Noの意味反転
          [はい、広告を受け取りたい]
          [いいえ、節約する気はありません]
          ユーザーの意図：
            「広告はいらない」
          UIが誘導する心理：
            「“節約しない人”になりたくない」
          ➡ 選択を歪めている

  文脈に応じて動的に変化する欺瞞的 UI の検知には限界があった
    概要
      文脈に応じて動的に変化する欺瞞的UIとは、ユーザーの操作・時間・状態に応じて表示内容や意味が変化し、
      表面的には正当だが、実際の行為を誤認させるUIである。

    根拠

        そして既存データセットの不十分な包括性について論じます
      
      https://arxiv.org/abs/2303.06782

        な手がかりを認識し、それらの検出、分類、および位置特定を可能にする、新しい自動化アプローチであるAidUI(画像＋テキストでも判定)
        評価の結果、\AidUIはダークパターンインスタンスの検出において、全体の適合率0.66、再現率0.67、F1スコア0.65を達成


    なぜ、自然言語処理（NLP）を用いると精度があがあるのか。

        

    何故自然言語処理＋動的 UI 検知が難しいのか
      AidUIより、「同じ種類のダークパターンでも多様な見た目を持つため、単純なルールでは検出が困難」
        →　同じ種類での多様性が挙げられる。

        1. デザインの自由度が高い
          同じ意図（誘導・誤認）でも色・配置・文言・画像を変えるだけで見え方が大きく変わる
            → ルールベースでは対応しづらい  
        2. 文脈依存性が強い
          ある文言やボタンがダークパターンか否かは、
          前後の文脈やユーザー行動履歴
          によって変わる場合がある
            → 静的に解析できない
        3. 動的生成・遅延表示がある
          JavaScript で動的に DOM を変更する UI（表示タイミングで誘導内容やラベルが変わる）
          → 静的スナップショットでは情報が欠ける

  他関連する要素
    はい、NLP 以外の方法も重要で、多く併用されます

    たとえば：

    ✔ コンピュータビジョン（画像として UI を解釈）
    ✔ 行動ログ解析（クリック/スクロール履歴を解析）
    ✔ 統計/機械学習モデルによるパターン検出
    ✔ ルールベースのヒューリスティック解析

    NLP は テキストの意味理解に強いですが、
    UI 解析全体では テキスト以外の情報（視覚/レイアウト/行動） も考慮する必要があるため、よく複数技術が組み合わされます。

  クラウド型AIによる解析は、通信環境による遅延の不安定さやプライバシー流出が、即時性を要する UI 介入において大きな障壁
    
    クラウド型AI
      インターネットを介してユーザーのデバイスからデータをクラウドサーバに送信し、
      クラウド上で AI モデルが推論（解析）を行った後、結果をユーザーに返す仕組み

        ・Web アプリからテキストや UI 情報を送る
        ・クラウド上の AI が解析（例：大規模言語モデル・NLP・分類モデルなど）
        ・結果を返す
        ・返ってきた結果を UI に反映する

    WebNN は W3C の標準化プロセス（候補段階）にあり、Chromium 系ブラウザでは実装が進んでいますが、
    安定版でデフォルトで有効になるという段階ではありません。実装にはフラグ有効化や開発版の利用が必要な例が多いです
    。従って、WebNN 利用は新規性がありながらも実装成熟度に大きなばらつきがある技術領域であり、実装と評価という意味
    で修士研究としての価値があります。

      目で認識( ≈ 100ms)　　            大体0.1秒　
      判断を伴う反応時間（200〜500 ms）  大体0.2 ~ 0.5秒
      意思決定を伴う反応（500 ms〜2 s）　大体0.5 ~ 2秒



  ⑥「数百 ms でも十分では？」への最終回答（これを言えると強い）

    数百 ms の遅延が常に保証されるなら問題ありません。
    しかし UI 介入は、一度でも間に合わなければ失敗する性質を持ちます。
    そのため、平均遅延ではなく、最悪遅延をゼロに近づける設計が求められます。
    これを満たせるのが、完全クライアントサイド推論です。

    　→つまり、従来のクラウド型推論とローカル推論でどの程度ダークパターンが防げるのか
    　　などの評価が必要になってくると考えられます。

WEBNNと類するAPI群の特徴と設計思想
  WebNN
    ブラウザ上で ニューラルネットワーク推論を標準 API で実行することを目指す仕様。
    WebGPU や WebAssembly と連携して、CPU／GPU／NPU 上で効率的に推論が可能になるインターフェースを提供します。
    → ブラウザ本体に 機械学習推論の命令セットを標準化しようとしているものです。
    → GPU や CPU を、低レベルで最適に使えるよう設計されています。
    → 仕様はまだ途中（W3C 勧告候補）のため、すべてのブラウザが完全対応していません。

    イメージ：
      WebNN は「Webブラウザ自身に AI 推論の仕組みをちゃんと組み込もうとしている標準案」
        → こうなると将来、どのブラウザでも同じ API で高速推論できるようになる可能性がある

  ONNX Runtime Web
    既存の AI モデル標準形式である「ONNX モデル」をそのままブラウザで動かす JavaScript ランタイムです。

    イメージ：
      ONNX モデル（AI を表現したファイルフォーマット）」を
       → そのまま JavaScript で読み込み・推論できるようにしたツール

    良いところ
      ✔ いろんなモデルをそのまま使える
      ✔ WebGPU や CPU をうまく使って推論できる（例えば Stable Diffusion Turbo など）

    注意点
      ❌ WebNN のような “ブラウザ標準 API” ではない
      → ライブラリとして動く形

  WebGPU + Transformers.js（例）

    → WebGPU は「ブラウザ内で GPU 演算を直接使うための低レベル API」です。
    → これ自体は AI 用ではなく、GPU 演算全般（3D描画・ML演算など）に使える APIです。
    → その上に Transformers.js のようなライブラリを載せることで ML 推論ができます。

    イメージ：
      WebGPU ＝ GPU に「命令」を投げられる仕組み
        → その上で独自の仕組み（Transformers.js など）が 「AIモデルを動かす」
        → 結果的にブラウザ上で LLM や画像モデルが動く



「WebNN は標準 API 候補ですが、現状ではブラウザやバックエンド（CPU/GPU/NPU）ごとに実装や性能が大きく異なります。他のブラウザ内推論技術（ONNX Runtime Web など）と比較して、
どの条件でユーザー体験（遅延・反応速度・精度）が改善されるかを 体系的に評価することがまだ十分に行われていません。
私の研究では、この比較評価と設計指針の整理を通じて、ブラウザ内 AI 推論の現実的な実装戦略とユーザー体験への影響を明らかにします。」

なぜ「自己防御型ブラウザ拡張基盤の開発と評価」を研究しようと思ったのか？
  現実のWebには意図しない行動を誘導する**ダークパターン（欺瞞的 UI）**が大量に存在し、ユーザー行動や意思決定に実害を与えるという実データおよび実験結果が多数報告されています。
単に検出するだけでなく「誤誘導を防ぐ仕組み（介入）」を構築・評価する研究はまだ未成熟であり、改善の社会的意味があります。

なぜ WebNN を使うのか？
  WebNNを使う理由は、ブラウザ内推論をハードウェアに近い形で実行できる可能性があり、
  低遅延UI介入への適性を検証する価値があるからです。
  
  WebNNは、ブラウザ内での学習済みモデルを使って判断・予測する動作を
  効率的に実行できるようにML専用に設計された低レベルのAPIです。
  現在はブラウザ標準化途中ですが、将来的にブラウザ内部の最適化と統合される可能性が高く、
  既存のWebGPUやWebAssembly経由の実装と比較して、リアルタイムのUI介入や応答性の影響を
  比較し検証する価値があると考えました。

    →　検証する価値がある？なぜ？
      低レベルAPIの違いが、UI介入の成功率やユーザー行動にどの程度影響を与えるのかは、まだ整理されていないからです。
      単なる推論速度比較ではなく、「どの構成が実際の誤操作防止に有効か」という観点
      で検証する点に価値があると考えています。
        つまり　「※※　　どの実行経路がUI介入に適しているかの実測比較」

        NPU有無は制御変数の一つです。
        まずは一般的なGPU/CPU環境で比較評価し、
        NPUが利用可能な環境では追加実験を行います。
        研究の本質はAPI経路の違いによる遅延差の定量評価です。

  WebNN を用いる理由は、ブラウザ内での学習済みモデルを使って判断・予測する動作を
  、将来的に最も効率的に実行できる可能性があるML専用に設計された低レベルのAPI だからです。モデルを実行する処理 を高速化で、
  現時点では標準化途中ですが、UI 介入のような低遅延が求められる用途において
　どの程度実用的かを検証する価値があると考えています。


  推論（Inference）」とは？

    ブラウザ内推論はこの「推論」を指す

Q5-a. 被験者実験の定量化はどうするのか？


なぜ研究価値があるのか？

既存研究では「ブラウザ内推論の性能評価」や「ダークパターン検出」はありますが、
それを「リアルタイム介入に結び付けて実装 → 定量評価」 まで行っている学術成果はほぼ見つかっていません。
→ つまり 「UI 介入効果の実証評価」という観点で新規性があります。


  具体的な定量指標

    ✔ 探索成功率
    ユーザーが目標情報に到達できた割合。

    ✔ クリック誤操作率
    誤誘導ボタンなどを押してしまった割合。

    ✔ 意思決定時間
    UI あり／なしで比較。

    ✔ 心理的負担スコア
    NASA-TLX などの心理尺度で評価。

    ✔ 視線追跡・生体データ
    optional だが、認知負荷を指標化可能。

    → これらは HCI / UX 研究で標準的に使われる定量手法で、「評価そのもの」は意味ある設計です。


  WebNN を含む複数のブラウザ内推論技術は、設計思想や実装方法が異なります。
  私の研究では、即時性が重要な UI 介入（例：誤誘導・欺瞞的 UI を防ぐ判断支援）という観点で、どの推論方式がより速く・効果的に動作するか を比較します。
  comparison は単なる性能ベンチマークではなく、ユーザー体験にどの程度寄与するかを観点にしています。」
